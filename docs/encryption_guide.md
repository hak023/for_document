# 서버 내 개인정보 및 과금 CDR 데이터 암호화 기술 가이드

본 문서는 서버 내에 저장되는 개인정보 및 과금 CDR(Call Detail Record) 데이터를 보호하고, 관련 법적 규제 및 보안 정책을 준수하기 위한 암호화 가이드라인입니다.

---

## 1. 개인정보 암호화 가이드

### 1.1 개인정보의 정의 및 암호화 대상
서버 내에 수집 및 저장되는 데이터 중 다음 항목들은 법적 규제 대상인 개인정보에 해당하며, 반드시 암호화하여 저장해야 합니다.

*   **식별 및 연락처 정보:** 이름, 휴대폰 번호, 주소 등
*   **통신 기록:** 통화 기록(Call Log), 문자 메시지 발송/수신 기록(SMS Log) 등

### 1.2 암호화 적용 방식

#### A. 데이터베이스 (DB) 암호화
*   DB에 저장되는 모든 개인정보 및 통신 기록은 **SAFEDB 솔루션**을 이용하여 암호화 처리를 수행해야 합니다.
*   SAFEDB의 세부 설정, API 호출 및 설치 방법에 대해서는 사내에 구비된 **별도의 SAFEDB 운영 가이드 문서**를 참조하시기 바랍니다.

#### B. 로그 (Log) 파일 암호화
*   서버 내에 생성되는 텍스트 로그 파일에 개인정보가 포함되는 경우, **24시간 이내에 평문 상태로 유지할 수 없습니다.** 즉, 당일 생성된 로그는 24시간 이내에 암호화되거나 파기되어야 합니다.
*   로그 파일 암호화 방식은 다음 두 가지 방안 중 시스템 환경에 적합한 방식을 선택하여 적용합니다.
    1.  **SAFEDB 키 관리 연동 AES-256 암호화:** SAFEDB 솔루션에서 제공하는 키 관리 모듈을 호출하여 대상 로그 파일을 대칭키 방식(AES-256)으로 암호화합니다.
    2.  **GPG(GNU Privacy Guard) 암호화:** 비대칭키(공개키/비밀키) 방식을 이용하여 로그 파일을 암호화합니다. 이 방식은 암호화 시에는 공개키만 사용하므로 키 탈취에 대한 보안성이 우수합니다.

## 2. 과금 CDR 암호화 가이드

과금 데이터(CDR)는 과금 방식(CTF, CDF)에 따라 상이한 보관 및 암호화 라이프사이클을 적용합니다.

### 2.1 CTF (Charging Trigger Function) 방식
CTF 방식은 실시간성 과금을 위해 데이터를 OFCS(Offline Charging System)로 즉시 전송하고, 로컬 서버에는 전송 이력만 백업 용도로 보관하는 형태입니다.

*   **동작 흐름:** CDR 생성 $\rightarrow$ OFCS 실시간 전송 $\rightarrow$ 서버 내 로컬 이력 저장
*   **암호화 방식:** 로컬 서버에 저장된 잔여 이력 파일은 즉시 암호화되어 보관되어야 합니다.
*   **기술 적용:** SAFEDB의 키 관리 API를 연동한 **AES-256** 방식 또는 **GPG 암호화** 방식을 적용하여 저장합니다.

### 2.2 CDF (Charging Data Function) 방식
CDF 방식은 로컬 서버에 과금 데이터를 보관하고 있으면, MZN 서버가 FTP(또는 SFTP) 프로토콜을 통해 배치(Batch) 형태로 가져가는 방식입니다. MZN 서버가 평문 데이터를 정상적으로 수집할 수 있도록 수집이 완료된 이후에 암호화 작업을 수행해야 합니다.

#### A. 처리 프로세스 및 백업 흐름
1.  **과금 파일 보관:** 지정된 CDR 생성 디렉토리에 평문 과금 파일이 보관됩니다.
2.  **MZN 서버 수집:** MZN 서버가 약속된 시간(예: **매일 새벽 2시**)에 FTP로 접속하여 해당 디렉토리의 평문 과금 파일을 수집해 갑니다.
3.  **백업 폴더 이동:** MZN 서버의 수집 작업이 완료된 시점(예: 새벽 2시 이후)에 기존 디렉토리의 과금 파일을 별도의 **암호화 백업 폴더**(`ex. /data/cdr/backup_secure`)로 이동합니다.
4.  **백업 데이터 암호화:** 백업 폴더로 이동된 과금 파일을 암호화하여 저장합니다.

#### B. 자동화 스케줄링 (Cron)
*   위의 이동 및 암호화 작업은 자동화 쉘 스크립트(`cdr_encrypt_backup.sh`)를 통해 수행하며, 리눅스 크론탭(Crontab)에 등록하여 주기적으로 실행합니다.
*   **크론탭 설정 예시 (매일 새벽 3시 실행):**
    ```bash
    0 3 * * * /usr/local/bin/cdr_encrypt_backup.sh > /dev/null 2>&1
    ```

#### C. 이슈 발생 시 복호화 및 복구 가이드
MZN 서버가 수집 장애 등을 일으켜 특정 시점의 CDR 데이터를 재수집해야 하는 경우, 다음 절차에 따라 복구합니다.
1.  **대상 백업 파일 복호화:** 백업 폴더 내 장애가 발생했던 시점의 암호화된 백업 파일을 복호화합니다.
2.  **원래 디렉토리로 이동:** 복호화된 평문 파일을 MZN 서버가 FTP로 수집하는 **원래의 CDR 디렉토리**로 복사(또는 이동)합니다.
3.  **재수집 수행:** MZN 서버 측에 재수집을 요청하거나, 다음 수집 주기에 MZN 서버가 해당 파일을 다시 가져갈 수 있도록 유도합니다.

## 3. GPG (GNU Privacy Guard) 키 관리 초안 가이드

GPG는 비대칭 키 시스템을 활용하여 데이터를 안전하게 암호화하고 복호화할 수 있는 강력한 오픈소스 도구입니다.

### 3.1 GPG 키 쌍(KeyPair) 생성
암복호화를 위해 먼저 비대칭 키(공개키 및 비밀키)를 생성합니다. 일반적으로 **암호화 서버**에는 공개키(Public Key)만 등록하고, 실제 복호화가 이루어지는 **보안/복구 서버**에만 비밀키(Secret/Private Key)를 위치시키는 것이 안전합니다.

```bash
# 키 쌍 생성 실행
gpg --full-generate-key
```
*키 생성 시 키 종류(RSA 권장), 키 크기(2048 또는 4096 bits), 유효기간, 사용자 정보(이름, 이메일) 및 Passphrase(비밀번호)를 설정해야 합니다.*

### 3.2 키 목록 확인
```bash
# 시스템에 등록된 공개키 목록 확인
gpg --list-keys

# 시스템에 등록된 비밀키 목록 확인
gpg --list-secret-keys
```

### 3.3 키 내보내기 및 가져오기 (배포/백업)
*   **공개키 내보내기 (암호화 대상 서버로 배포 목적):**
    ```bash
    gpg --export --armor -o recipient_pubkey.asc [사용자 이메일 또는 Key ID]
    ```
*   **비밀키 내보내기 (안전한 보관 및 백업 목적 - 절대 외부 유출 금지):**
    ```bash
    gpg --export-secret-keys --armor -o secret_key.asc [사용자 이메일 또는 Key ID]
    ```
*   **키 가져오기 (대상 서버에서 실행):**
    ```bash
    gpg --import recipient_pubkey.asc
    ```

### 3.4 GPG 파일 암복호화 명령어

#### A. 파일 암호화
MZN 백업 폴더 내의 파일을 수신자의 공개키를 사용하여 암호화합니다.
```bash
# 파일 암호화 명령어
gpg --encrypt --recipient [수신자 이메일] --output [암호화할파일명.gpg] [원본파일명]

# 예시:
gpg --encrypt --recipient charge_admin@company.com --output billing_20260616.dat.gpg billing_20260616.dat
```

#### B. 파일 복호화 (장애 복구 시 사용)
비밀키를 보유한 담당자가 비밀번호(Passphrase)를 입력하여 평문으로 변환합니다.
```bash
# 파일 복호화 명령어
gpg --decrypt --output [평문복원파일명] [암호화된파일명.gpg]

# 예시:
gpg --decrypt --output billing_20260616.dat billing_20260616.dat.gpg
```

## 4. GPG 키 관리 및 보안 권장사항

1.  **비밀키의 격리:** 데이터를 암호화하기만 하는 일반 프로덕션 서버에는 공개키(`Public Key`)만 등록하십시오. 복호화에 필요한 비밀키(`Secret Key`)는 인가된 관리자 PC 또는 별도의 복호화 보안 서버에만 보관해야 서버가 해킹당하더라도 데이터 누출을 방지할 수 있습니다.
2.  **Passphrase 안전 보관:** 비밀키를 생성할 때 입력한 Passphrase(암호)는 분실 시 복구가 절대 불가능하므로, 오프라인 망이나 사내 자격증명 관리 시스템(Vault 등)에 백업해 두어야 합니다.
3.  **키 만료 관리:** 키 생성 시 만료일(Expiration Date)을 설정하는 것이 좋으며, 만료 전에 키 롤오버(갱신 및 재교체) 프로세스를 수립해야 합니다.
